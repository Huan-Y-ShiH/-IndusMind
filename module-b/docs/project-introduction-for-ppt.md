## 0. 阅读说明：项目能力状态如何理解

为了避免汇报时把原型能力描述成生产能力，本文统一使用以下状态：

- **已实现**：仓库中已有代码，并有测试、脚本或实际链路可验证。
- **可演示**：可以完成端到端演示，但数据规模、并发、稳定性或工程化程度仍属于原型阶段。
- **降级可用**：外部模型或 Dify 不可用时，系统仍能依靠本地知识和规则给出受限结果。
- **占位未接入**：接口和安全约束已经定义，但没有真实 SCADA、波形平台或企业生产数据。
- **规划项**：适合后续升级，不应在当前 PPT 中表述为“已经完成”。

当前项目最准确的一句话定位是：

> **IndusMind 是一个面向航空发动机与燃气轮机预测性维护的可解释诊断 Agent：它接收上游异常检测或 RUL 模型输出，通过“本地结构化知识 + Dify 语义知识 + Qwen/DeepSeek 推理”形成分层根因、证据链、维修建议和诊断报告，并通过确定性工作流与人工门禁控制大模型风险。**

# 第一部分：项目概览

## 1. 项目名称与定位

### 1.1 项目名称

**IndusMind — 基于 RAG 与多模型协作的工业运维诊断 Agent**

英文描述：

> Industrial O&M Diagnostic Agent with RAG for CMAPSS-compatible turbofan prognostics.

### 1.2 项目定位

IndusMind 位于预测性维护系统的“诊断与决策中枢”位置。它不负责直接训练 RUL 模型，也不直接控制设备，而是把上游算法给出的异常分数、剩余寿命和特征归因，转化为运维人员能够理解和执行的内容：

1. **发生了什么异常**；
2. **最可能的根因是什么**；
3. **根因位于哪个系统、部件或子模式**；
4. **有哪些证据支持或反驳该结论**；
5. **是否需要人工复核**；
6. **建议采取什么维修或监测行动**；
7. **能否形成结构化结果和完整诊断报告**。

### 1.3 在三模块系统中的位置

如果完整系统按 A/B/C 三个模块划分：

- **Module A：监测与预测引擎**
  - 输入传感器时间序列；
  - 输出异常类型、异常分数、RUL、RUL 曲线、特征归因等。
- **Module B：IndusMind Agent 引擎**
  - 负责知识检索、故障诊断、证据验证、维修决策、报告生成；
  - 本仓库即 Module B。
- **Module C：前端 / 网关 / 业务系统**
  - 展示诊断任务与结果；
  - 管理设备、用户、工单和交互；
  - 通过异步 HTTP API 调用 IndusMind。

IndusMind 与上下游解耦：

- 上游字段通过 `DiagnoseJobRequest` 适配为内部 `AnomalyEvent`；
- 下游通过提交任务与轮询结果获取结构化诊断；
- Server 酱通知属于旁路能力，不依赖 Module C，也不会阻塞主诊断流程。

## 2. 项目背景与行业痛点

### 2.1 从“预测故障”到“解释故障”的缺口

传统预测性维护模型通常擅长回答：

- 设备是否异常；
- 异常分数是多少；
- 剩余寿命还有多少；
- 哪些传感器对预测贡献较大。

但真实运维决策还需要回答：

- 为什么这些传感器会一起变化；
- 它们对应压气机、风扇、涡轮还是燃烧系统；
- 是效率退化、流量退化、结垢、磨损还是复合故障；
- 当前结果是数据标签、文献推断、种子案例还是实际维修证据；
- 应立即停机、七日内计划维修，还是继续监测；
- 当模型置信度不足时，系统是否会拒绝自动维修并转人工。

IndusMind 解决的正是“预测结果到运维知识与行动”之间的解释鸿沟。

### 2.2 工业知识的异构性

工业诊断知识不是单一形态：

- FMEA 是结构化表格，适合精确匹配；
- 维修工单包含设备、故障分类和历史动作；
- 手册、论文和案例是自然语言文本；
- 传感器元信息是编号到物理含义的映射；
- RUL 和特征归因来自模型，是事件级动态数据。

如果把所有数据都放入向量数据库，会损失 FMEA 的方向匹配精度和工单的精确查询能力；如果全部用本地规则，又无法理解自然语言机理。因此项目采用“双腿式 RAG”：

- **结构化知识留在本地精确查询**；
- **叙述性知识进入 Dify 做语义检索**。

### 2.3 大模型在工业场景中的风险

工业诊断不能只追求“回答流畅”，还必须控制：

- 幻觉根因；
- 无来源的维修建议；
- 把占位工具返回值当成真实证据；
- 把合成案例当成真实维修案例；
- 低置信度情况下自动下发维修；
- 外部 API 不可用后整条链路失效；
- 多 Agent 自主运行导致流程不可复现、不可评估。

IndusMind 的核心设计原则是：

> **确定性流程负责边界与门禁，RAG 提供证据，LLM 负责受约束的重排和解释，多专家 Crew 只在适合探索的 L2/L3 环节工作。**

## 3. 项目目标

### 3.1 核心目标

1. 将上游异常检测 / RUL 输出统一成标准异常事件；
2. 基于传感器编号和异常方向检索故障知识；
3. 输出 L1、L2、L3 分层根因；
4. 建立“征兆 → 候选 → 机理 → 结论”的逻辑路径；
5. 将维修工单历史转化为行动建议；
6. 对低置信结果自动转人工；
7. 在 Dify、Qwen 或 DeepSeek不可用时保持受限可用；
8. 通过异步 API 支持云端前端调用本地 Agent；
9. 通过测试和离线评估量化检索质量；
10. 为后续接入真实数据、SCADA 和波形平台预留扩展点。

### 3.2 当前明确不做

- 不负责训练上游 RUL / 异常检测模型；
- 不直接连接或控制真实设备；
- 不自动执行维修工单；
- 不使用 LangGraph 或 Hermes Agent 作为核心编排；
- 不把 FMEA、工单和传感器元信息全部向量化；
- 不把 `available=false` 的 SCADA / 波形占位输出视为证据；
- 不把种子、推断或合成案例包装成真实现场案例；
- 不在公开文档、日志或测试中保存真实密钥。

# 第二部分：领域与数据基础

## 4. NASA CMAPSS 场景

### 4.1 为什么选择 CMAPSS

NASA CMAPSS 是航空发动机预测性维护研究中常用的公开模拟数据体系，适合项目完成以下工作：

- 建立从退化趋势到故障模式的映射；
- 使用统一的 21 个传感器通道；
- 验证 RUL 异常、特征归因与诊断之间的衔接；
- 构建可以公开复现的冷启动知识库；
- 避免依赖难以公开获取的整机厂原始 FMEA。

### 4.2 数据子集与故障模式

- **FD001**：单工况，高压压气机 HPC 退化；
- **FD002**：多工况，高压压气机 HPC 退化；
- **FD003**：单工况，高压压气机 + 风扇复合退化；
- **FD004**：多工况，高压压气机 + 风扇复合退化。

公开 CMAPSS 文献通常通过 Fan、LPC、HPC、HPT、LPT 的效率、流量或压比健康参数注入故障。IndusMind 将这些健康参数变化整理为“故障模式 → 传感器 → 异常方向”的知识签名，用于 RAG 和 FMEA 匹配。

### 4.3 统一传感器编号

项目严格采用 NASA Saxena Table 2 的 `s1`—`s21`：

| 编号 | 物理量 |
|---|---|
| `s1` | 风扇进口总温 T2 |
| `s2` | 低压压气机出口总温 T24 |
| `s3` | 高压压气机出口总温 T30 |
| `s4` | 低压涡轮出口总温 T50，作为 EGT 代理 |
| `s5` | 风扇进口压力 P2 |
| `s6` | 外涵道总压 P15 |
| `s7` | 高压压气机出口总压 P30 |
| `s8` | 风扇物理转速 Nf |
| `s9` | 核心机物理转速 Nc |
| `s10` | 发动机压比 epr |
| `s11` | 高压压气机出口静压 Ps30 |
| `s12` | 燃油流量与 Ps30 之比 phi |
| `s13` | 风扇换算转速 NRf |
| `s14` | 核心机换算转速 NRc |
| `s15` | 涵道比 BPR |
| `s16` | 燃烧室燃油空气比 farB |
| `s17` | 引气焓 htBleed |
| `s18` | 需求风扇转速 Nf_dmd |
| `s19` | 需求换算风扇转速 PCNfR_dmd |
| `s20` | 高压涡轮冷却引气 W31 |
| `s21` | 低压涡轮冷却引气 W32 |

项目曾存在文档示例编号与 NASA 标准不一致的问题，当前已经通过元信息文件和别名映射统一。PPT 中必须使用以上定义，不能把 `s11`、`s14`、`s15`误写成 EGT、燃油流量或压比。

### 4.4 方向化征兆

仅知道“哪个传感器重要”还不够，同一传感器升高和降低可能对应完全不同的故障。项目将征兆表示为：

```text
s3↑;s4↑;s12↑
```

内部统一为：

```json
[
  {"feature": "s3", "direction": "high"},
  {"feature": "s4", "direction": "high"},
  {"feature": "s12", "direction": "high"}
]
```

FMEA 和案例匹配同时比较：

- 传感器编号；
- 异常方向；
- 可选的贡献度或偏离强度。

这是项目区别于“只按关键词检索”的关键技术点。

## 5. 当前知识资产

截至核验日期，本地冷启动知识包括：

- 30 条 FMEA 记录，覆盖 CMAPSS-Turbofan 22 条、TAURUS-60 5 条、CFM56-7B 3 条；
- 5 组 L1 故障展开；
- 10 个 L2 子模式；
- 3 个种子 / 数据集标签案例；
- 4 个手册或文献知识 chunk；
- 8 条种子维修工单；
- 21 个 CMAPSS 传感器物理定义；
- NASA、N-CMAPSS、燃气轮机 FMEA、SKF 轴承手册等来源材料。

### 5.1 证据等级

系统对知识来源进行区分：

- `literature`：NASA、SKF 等公开文献；
- `dataset_label`：数据集定义或标签；
- `seed`：冷启动种子数据；
- `inferred`：由气路物理或文档类比推断；
- `unverified`：尚未充分核实。

合成案例会降低匹配权重，不能与真实维修历史等权。当前 3 个案例均属于冷启动或数据集标签性质，因此只能用于演示、检索和算法验证，不能作为自动维修依据。

### 5.2 典型故障签名

**HPC 退化常见征兆：**

- T30（`s3`）升高；
- T50 / EGT 代理（`s4`）升高；
- phi（`s12`）升高；
- 核心机转速 Nc / NRc 可能升高；
- P30 可能下降；
- 热端和冷却相关通道可能漂移。

**Fan 退化附加征兆：**

- Nf / NRf 异常，常见升高；
- BPR 下降；
- 为维持推力，热端温度可能升高。

这些方向并非在所有工况中绝对成立。闭环控制、工况变化和传感器弱敏感通道会影响方向，因此系统保留证据等级和人工门禁。

# 第三部分：总体技术架构

## 6. 架构总览

IndusMind 采用四层业务架构和多层技术支撑：

```text
上游监测 / RUL 模型
        │
        │ 异常分数、RUL、特征归因、方向
        ▼
┌────────────────────────────────────────────┐
│ L1 监测层：事件标准化与语义化              │
│ AnomalyEvent → DiagnosticQuery              │
└────────────────────────────────────────────┘
        │
        ▼
┌────────────────────────────────────────────┐
│ L2 诊断层：混合 RAG + LLM 重排 + 专家细化   │
│ Dify / 本地案例 / FMEA / 手册 / Crew        │
└────────────────────────────────────────────┘
        │
        ▼
┌────────────────────────────────────────────┐
│ L3 决策层：工单历史 + 维修建议 + 紧急度      │
└────────────────────────────────────────────┘
        │
        ▼
┌────────────────────────────────────────────┐
│ L4 报告层：结构化结果 + Markdown + 旁路通知  │
└────────────────────────────────────────────┘
        │
        ├── 云端前端异步轮询
        └── Server 酱旁路通知
```

技术支撑层包括：

- CrewAI Flow；
- Pydantic State 与结构化 Schema；
- CrewAI Crew；
- Qwen / DeepSeek 双模型路由；
- Dify Retrieval API；
- 本地 FMEA、案例、工单和元信息；
- FastAPI 异步任务服务；
- frp + ECS 公网中转；
- pytest 与离线评估脚本。

## 7. 为什么核心编排选择 CrewAI Flow

### 7.1 核心诉求

工业诊断流程必须：

- 顺序明确；
- 状态可追踪；
- 分支条件可解释；
- 失败行为可预测；
- 能被单元测试与离线评估；
- 不能由大模型随意决定是否跳过安全步骤。

因此项目使用 CrewAI Flow 的：

- `@start`：定义起点；
- `@listen`：定义确定性步骤依赖；
- `@router`：定义置信度门禁分支；
- Pydantic State：保存每个阶段的结构化状态。

### 7.2 为什么不使用通用自主 Agent 作为核心

LangGraph、Hermes 或通用自主 Agent 并非不能完成任务，但本项目更强调：

- 固定的监测—诊断—决策—报告路径；
- 可重复评估；
- 低置信度强制转人工；
- 工具不可用时明确降级；
- LLM 不得绕过证据和门禁。

因此核心状态机由 Flow 控制，而不是把整个系统交给自主 Agent。

### 7.3 Flow 状态

`DiagnosticFlowState` 保存：

- `event`：标准异常事件；
- `query`：诊断检索查询；
- `rag_results`：混合检索候选；
- `dify_explain_hits`：弱匹配时的 Dify 解释材料；
- `l1_diagnosis`：L1 候选根因；
- `l2l3_diagnosis`：细化诊断；
- `decision`：行动方案；
- `report`：最终 Markdown 报告；
- `need_human_review`：人工复核标志；
- `retry_count`：重试状态预留。

## 8. 完整执行流程

### 8.1 步骤 1：监测输入

Flow 接收 `AnomalyEvent`。典型字段包括：

- 事件 ID；
- 设备 ID 与型号；
- 时间戳；
- 上游模型版本；
- 异常类型与异常分数；
- 时间窗口；
- 特征归因；
- RUL 预测值与序列；
- RUL 形态、曲率、残差和漂移；
- 仅有 RUL 时的伪归因；
- 原始数据引用。

### 8.2 步骤 2：事件语义化

系统将结构化事件转换成 `DiagnosticQuery`：

- 设备型号；
- 异常类型；
- Top 特征；
- 方向化征兆；
- 简短征兆文本；
- 适合语义检索的自然语言查询。

这一步建立了模型字段与知识库语言之间的桥梁。

### 8.3 步骤 3：混合检索

系统并行或顺序获取：

1. Dify 案例语义命中；
2. 本地案例方向化 Jaccard 命中；
3. 本地 FMEA 方向化 Jaccard 兜底；
4. Dify 手册文本佐证。

案例和 FMEA 用于形成“根因候选”，手册用于提供机理证据，不允许手册向量分直接挤掉根因候选。

### 8.4 步骤 4：L1 根因重排

系统将最多 10 条安全裁剪后的候选交给快速重排模型：

- 主通道：Qwen Plus；
- 备用通道：DeepSeek Chat。

输出包括：

- 根因候选；
- 置信度；
- 匹配案例；
- 证据；
- 机理；
- 逻辑路径；
- 是否需要人工复核。

为了防止 LLM 夸大置信度，正常模式下候选置信度会被 RAG 支持分上限约束。

### 8.5 步骤 5：置信度门禁

正常生产语义下，高置信路径需要同时满足：

- Top-1 置信度大于 `0.8`；
- 对应 RAG 支持分不低于 `0.5`；
- L1 结果没有主动要求人工复核。

否则系统进入人工复核分支：

- 保留最可能的 L1 候选；
- 标记 `need_human_review=true`；
- 生成“保持监测并转人工诊断”的建议；
- 禁止依据当前候选自动维修；
- 仍然输出报告，避免前端只得到空结果。

### 8.6 步骤 6：L2/L3 专家细化

高置信路径进入 L2/L3 细化。这里采用“确定性 Python 预处理 + CrewAI Crew 评审综合”的实现：

1. Python 先调用本地 `expand_fmea`，确定性生成 L2 子模式；
2. Python 再通过 `asyncio.gather`，为每个假设并行预采集工况、维修历史、波形和二次 RAG 证据；
3. CrewAI 的**故障假设生成专家**检查给定假设的完整性和竞争关系，不新增知识库外失效模式；
4. **假设验证专家**评估预采集的 JSON 证据，并忽略 `available=false` 的工具结果；
5. **根因综合专家**使用复杂推理模型综合竞争假设，输出 L2/L3、位置、严重度、证据链、反证和维修建议。

Crew 内部任务按 Hypothesize → Verify → Synthesize 顺序执行。当前三个 Agent 并未直接绑定 CrewAI `tools=`；工具调用发生在进入 Crew 前的 Python 预处理阶段。这种方式使外部数据访问保持确定性，也便于测试和过滤不可用证据。

### 8.7 步骤 7：决策

系统根据诊断结论映射故障大类，并查询对应设备的历史维修工单：

- 如果 L2/L3 给出维修建议，优先采用；
- 如果没有建议但命中历史工单，提取历史动作；
- 如果仍无动作，正常模式转人工，演示模式给出继续监测与复核建议。

紧急度统一为：

- `immediate`；
- `planned_within_7d`；
- `monitor`。

### 8.8 步骤 8：报告

最终 Markdown 报告包含：

- 设备与事件；
- 异常类型和主要征兆；
- L1 / L2 / L3 诊断；
- 置信度；
- 证据说明；
- 故障机理；
- 逐步逻辑路径；
- 处置建议；
- 紧急度；
- 参考工单；
- 弱匹配时的 Dify 解释依据；
- 是否处于强制 Top-1 演示模式。

### 8.9 步骤 9：旁路通知

当 `NOTIFY_ENABLED=true` 时，Flow 完成后可通过 Server 酱推送：

- 设备；
- 根因；
- 置信度；
- 紧急度；
- 是否需要人工；
- 处置建议；
- 逻辑路径；
- 报告摘要。

通知失败只记录日志，不影响主流程；同一进程内按 `event_id` 去重。

# 第四部分：RAG 与知识工程

## 9. 双腿式知识架构

### 9.1 Dify 语义知识库

适合存放：

- 案例的故障叙述；
- 根因与机理文本；
- 手册段落；
- 论文摘要；
- 文献解释。

检索方式：

- Dify Retrieval API；
- 语义 / 向量检索；
- 通过 `case_id:` 或 `chunk_id:` 首行标记回查本地结构化数据。

### 9.2 本地结构化知识

适合存放：

- FMEA CSV；
- FMEA L2 展开 JSON；
- 维修工单 JSON；
- 传感器元信息；
- 案例结构化字段；
- 手册元数据。

检索方式：

- Python 精确过滤；
- 设备型号匹配；
- 故障分类匹配；
- 传感器编号 + 方向 Jaccard；
- 证据等级加权；
- 风险先验兜底。

### 9.3 为什么不全部上传 Dify

FMEA 和工单需要精确语义：

- `s4↑` 与 `s4↓` 不能被当成相似文本；
- `device_id` 需要精确匹配；
- 故障分类和工单动作需要确定性过滤；
- RPN、严重度、发生度等数值需要保留结构；
- 向量相似度不适合作为唯一依据。

因此 Dify 负责“理解文本”，本地代码负责“执行结构化约束”。

## 10. 混合检索融合算法

### 10.1 数据源

- Dify 案例：权重 `1.0`；
- 本地案例：权重 `0.95`；
- FMEA：权重 `0.85`；
- 手册佐证：权重 `0.55`。

### 10.2 分数融合

不同数据源的原始分数不可直接比较。系统进行分源归一化与排名修正：

```text
融合分 = 数据源权重 × (0.7 × 原始分 + 0.3 × 排名因子)
排名因子 = 1 / (1 + 0.15 × (rank - 1))
```

随后按根因去重并排序：

- 根因候选优先保留；
- 手册最多补充 3 条；
- 默认总候选最多 10 条。

### 10.3 本地 Jaccard

征兆集合例子：

```text
查询 = {s3_high, s4_high, s12_high}
FMEA = {s3_high, s4_high, s12_high, s9_high}
```

Jaccard：

```text
J(A,B) = |A∩B| / |A∪B|
```

在此基础上叠加来源权重：

- NASA / SKF 文献：`1.0`；
- seed：`0.75`；
- 推断：`0.65`；
- 未验证：`0.7`。

### 10.4 弱匹配解释机制

当开启演示模式且本地匹配分低于 `0.35` 时：

1. 取事件自然语言描述；
2. 加入 Top 候选根因；
3. 再次查询 Dify；
4. 使用命中的文献段落补充 mechanism、evidence 和 logic_path；
5. 在报告中显式列出 Dify 解释来源。

这允许系统在演示场景始终形成可读结论，但报告会标记模式，不能把它等同于生产高置信诊断。

### 10.5 Dify 故障降级

当 Dify：

- 未配置；
- 网络错误；
- API 返回异常；
- 暂时不可用；

系统会：

- 记录警告；
- 在 30 秒冷却期内跳过重复失败调用；
- 使用本地案例与 FMEA；
- 保持诊断主流程不被阻塞。

# 第五部分：LLM 与多专家协作

## 11. Qwen / DeepSeek 双通道

### 11.1 任务路由

| 任务 | 主模型 | 备用模型 |
|---|---|---|
| 结构化抽取 | Qwen Turbo | DeepSeek Chat |
| 快速候选重排 | Qwen Plus | DeepSeek Chat |
| L2/L3 深度因果推理 | DeepSeek Reasoner | Qwen Plus |
| 报告生成 | Qwen Plus | DeepSeek Chat |

### 11.2 选择原因

- Qwen 适合中文结构化抽取、快速重排和常规报告；
- DeepSeek Reasoner 适合多假设、因果链和反证综合；
- 双通道可以降低单一供应商超时、限流、网络故障的影响；
- 两者通过 OpenAI 兼容接口和 CrewAI / LiteLLM 接入，便于替换。

### 11.3 降级策略

- 主模型失败后自动调用备用模型；
- 主备均失败时抛出统一 `LLMError`；
- L1 失败则回退为 RAG 分数直排；
- L2/L3 失败则降级为 L1，不阻塞决策和报告；
- 无任何 API Key 时，系统仍可用本地 RAG 形成受限结果。

### 12.1 提示词防护

系统将检索内容明确描述为：

### 12.2 最小化外发数据

# 第六部分：对外 API 与前端集成

## 13. 为什么采用异步任务 API

完整诊断包含：

- RAG 检索；
- 模型调用；
- 多专家推理；
- 工具验证；
- 报告生成。

这些步骤可能耗时数秒到一分钟。若使用同步 HTTP 长连接：

- 容易被前端、网关或代理超时；
- 难以展示进度；
- 重试可能重复执行任务；
- LLM 波动会直接影响请求连接。

因此采用：

```text
POST 提交任务 → 202 + job_id
后台执行诊断
GET 轮询任务 → queued / running / succeeded / failed
```

### 13.1 提交接口

```text
POST /api/v1/diagnose/jobs
```

最低可运行字段：

- `event_id`；
- `device_id`；
- `timestamp`；
- `model_version`；
- `anomaly_type`；
- `rul_hours`。

推荐同时提供：

- `anomaly_score`；
- `rul_series`；
- `risk_level`；
- `feature_attribution`；
- 或 `pseudo_attribution`。

### 13.2 查询接口

```text
GET /api/v1/diagnose/jobs/{job_id}
```

任务状态：

- `queued`：进度 0；
- `running`：进度 50；
- `succeeded`：进度 100；
- `failed`：进度 100，并返回错误信息。

### 13.3 结果内容

成功结果包含：

- 事件、设备与输入透传字段；
- 根因；
- L1 / L2 / L3；
- 置信度；
- 是否人工复核；
- 逻辑路径；
- 机理与证据；
- 证据等级；
- 紧急度；
- 行动计划；
- 关联工单；
- Markdown 报告；
- RAG 命中数量。

### 13.4 API 工程能力

已实现：

- `X-Api-Key` 鉴权；
- Pydantic 参数校验；
- 统一响应包；
- CORS 与 OPTIONS 预检；
- `event_id` 幂等；
- 单并发诊断；
- 最多 3 个活跃任务；
- 超限返回 `429 busy`；
- 健康检查；
- 结构化错误码。

### 13.5 当前 API 限制

- 任务表保存在进程内存；
- 服务重启后任务记录丢失；
- 当前不使用 Redis、数据库或消息队列；
- 单实例最多执行 1 个 Flow；
- 回调字段为预留，本期忽略；
- 适合联调和演示，不是高可用生产任务平台。

# 第七部分：部署与远程调用

## 14. 当前部署拓扑

```text
云端前端 / 前端后端
        │
        │ HTTP API
        ▼
阿里云 ECS 公网中转
  frps + systemd
        │
        │ frp TLS 控制连接
        ▼
本地 macOS
  frpc + launchd
        │
        ▼
FastAPI :8002
        │
        ▼
CrewAI Flow + Dify + Qwen/DeepSeek + 本地知识
```

### 14.1 当前稳定性措施

- ECS 上 `frps` 由 systemd 守护；
- macOS 上 Agent API 和 `frpc` 由 launchd 守护；
- 进程退出后自动重启；
- 提供一键恢复脚本；
- 本地和公网均提供 `/health`；
- frp 控制端与业务映射端口分离；
- 真实 frp Token 不写入仓库。

### 14.2 为什么 ECS 只做中转

当前 Agent、Dify 访问和本地知识仍在开发机上，ECS 只提供稳定公网入口：

- 不需要立即迁移本地知识库；
- 不改变现有 Dify 连接；
- 前端不必与本地网络直接互通；
- 比临时 Cloudflare Quick Tunnel 拥有更稳定的固定入口。

### 14.3 当前部署风险

- 本地电脑关机或休眠时服务不可用；
- 当前公网业务入口为 HTTP；
- HTTPS 页面直接调用 HTTP 会触发浏览器混合内容拦截；
- 应由前端后端同源转发，或后续配置域名与 HTTPS 反向代理；
- 公网 IP、端口和 API Key 不应出现在公开 PPT；
- 若面向真实生产，需要将 Agent 容器化部署到长期在线环境。

# 第八部分：可靠性、安全与可解释性

## 15. 多层失败兜底

| 故障点 | 系统行为 |
|---|---|
| Dify 未配置或网络失败 | 回退本地案例和 FMEA |
| Qwen 主模型失败 | 切换 DeepSeek |
| DeepSeek 主模型失败 | 切换 Qwen |
| L1 LLM 输出解析失败 | 使用 RAG 分数直排 |
| L2/L3 Crew 超时或失败 | 降级为 L1 结论 |
| SCADA / 波形未接入 | 返回 `available=false`，不作为证据 |
| 置信度或证据不足 | 生成报告并转人工 |
| Server 酱推送失败 | 记录日志，不影响主流程 |
| 本地 Agent / frpc 退出 | launchd 自动重启 |
| ECS frps 退出 | systemd 自动重启 |

### 15.1 超时

- 整体 Flow 默认 90 秒；
- L2/L3 Crew 默认 60 秒；
- Server 酱请求 15 秒；
- 前端建议 2—5 秒轮询一次；
- 前端总等待建议不超过 120 秒。

### 15.2 人工门禁

默认模式下，系统不承诺“任何输入都必须给结论”。如果证据不足：

- 输出候选；
- 标记人工复核；
- 建议继续监测；
- 禁止自动维修。

`DIAGNOSTIC_ALWAYS_TOP1=true` 仅用于：

- 联调；
- 演示；
- UI 展示；
- 验证逻辑链与 Dify 解释能力。

PPT 中必须说明：**始终 Top-1 模式不是生产安全策略。**

---

## 16. 可解释性设计

IndusMind 的可解释性不是只让 LLM“写一段理由”，而是分层建立：

1. **输入解释**
   - 哪些传感器贡献最大；
   - 异常方向是什么；
   - RUL 如何变化。
2. **检索解释**
   - 命中了哪个案例或 FMEA；
   - 匹配分是多少；
   - 来源属于文献、标签、种子还是推断。
3. **机理解释**
   - 为什么该故障会导致这些温度、压力、转速变化。
4. **推理路径**
   - 观测征兆；
   - 候选失效模式；
   - 机理验证；
   - 最终根因。
5. **决策解释**
   - 为什么给出该紧急度；
   - 是否命中历史工单；
   - 为什么需要人工复核。
6. **反证能力**
   - L2/L3 结构保留 `counter_evidence`；
   - 可以记录被拒绝假设及原因。

## 20. 技术创新点

# 第十五部分：可直接用于摘要、申报书和答辩的文案

## 28. 100 字摘要

IndusMind 是面向航空发动机与燃气轮机预测性维护的工业诊断 Agent。系统接收异常检测与 RUL 模型输出，结合本地 FMEA、维修工单、Dify 语义知识及 Qwen/DeepSeek 推理，生成分层根因、证据链、维修建议和报告，并通过确定性 Flow、人工门禁和多级降级保障诊断可控性。

## 29. 300 字摘要

IndusMind 面向航空发动机和燃气轮机预测性维护场景，重点解决“上游模型能发现异常，却难以给出根因解释与维修行动”的问题。系统以 NASA CMAPSS 为公开数据基础，将异常分数、RUL、RUL 曲线和传感器特征归因标准化为诊断事件。核心流程使用 CrewAI Flow 构建监测、诊断、决策和报告四层确定性状态机；知识检索采用“双腿式 RAG”，将 FMEA、工单和传感器元信息保留在本地进行方向感知的精确匹配，将案例叙述和手册文本同步至 Dify 进行语义检索。Qwen 与 DeepSeek 按任务类型路由并互为备用，L2/L3 环节通过多专家 Crew 完成假设生成、证据验证和根因综合。系统默认在证据不足时转人工，禁止依据低置信候选自动维修，并支持异步 API、云端前端调用、Markdown 报告和 Server 酱旁路通知。当前原型已通过 44 项自动化测试，冷启动评估集 Top-3 召回为 1.0，具备端到端演示能力。

## 30. 答辩开场稿

传统 RUL 模型可以告诉我们一台发动机还剩多少寿命，但它通常不能直接告诉运维人员：为什么寿命下降、哪个部件正在退化、有哪些证据、下一步应该做什么。IndusMind 的目标，就是把预测模型的数值结果转化为可解释、可追溯、可执行的运维诊断。我们没有把整个系统交给一个自由运行的大模型，而是用确定性 Flow 控制监测、诊断、决策和报告，用本地 FMEA 与 Dify 语义知识提供证据，用 Qwen 和 DeepSeek 完成受约束的重排与推理，并在证据不足时强制转人工。这使系统既保留了大模型的表达与推理能力，也满足工业场景对安全、可控和可评估的要求。

## 31. 答辩结尾稿

IndusMind 当前完成的是一个能够端到端运行的诊断 Agent 原型：它已经打通上游异常事件、混合知识检索、分层诊断、维修建议、报告、通知和前端异步调用。更重要的是，项目从一开始就把证据等级、失败降级、人工门禁和工程边界纳入设计。下一阶段，我们将以真实设备数据和维修闭环为核心，扩充案例与标注，接入 SCADA 和波形工具，并将内存任务服务升级为可持久化、可扩展的生产架构。最终目标不是让大模型替代工程师，而是让工程师更快地获得有证据、有逻辑、可复核的诊断支持。

## 结论

IndusMind 的技术主线可以概括为：

> **标准化异常事件提供入口，方向感知混合 RAG 提供知识依据，Qwen/DeepSeek 提供受约束的重排与推理，CrewAI Flow 保证确定性，CrewAI Crew 完成局部多专家协作，人工门禁与多级降级保证工业安全，异步 API 与 frp 完成前后端工程连接。**

当前项目已经具备完整原型和端到端演示能力，但仍应诚实定位为“冷启动、可验证、可扩展的工业诊断 Agent 原型”。后续工作的关键不是继续堆叠更多 Agent，而是引入更多真实闭环数据、严格知识治理、持久化任务基础设施和生产级安全部署。
